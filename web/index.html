Design Rationale & Market Comparison
Why a layered Kamailio-fronted architecture wins on security and scale against common market approaches.
Approach Security posture Scale & HA Operational complexity Fraud protection WebRTC support
Layered Kamailio + RTPengine + Asterisk ✓ Our choice Edge absorbs floods before reaching call engine. TLS/SRTP throughout. Topology hidden. Stateless edge scales horizontally behind anycast. Asterisk pool behind dispatcher. Higher initial setup - offset by infrastructure-as-code and clear layer separation. Per-account caps, destination allow-lists, velocity limits, auto-suspend at dialplan level. RTPengine bridges DTLS-SRTP natively. WSS at edge. No additional gateway needed.
Asterisk / FreePBX direct on public IP Asterisk exposed to internet. All attacks hit the B2BUA directly. High SIP scanning exposure. Single point of failure. Asterisk under flood will exhaust threads and drop calls. Simple single-host setup. Basic dialplan guards only. No edge filtering. Limited. Requires additional WebRTC proxy configuration.
Single-box commercial PBX Vendor-managed hardening. Limited visibility. Proprietary SBC often shared. Vertical scaling only. No horizontal elasticity. Low - vendor managed. Basic built-in rules. No custom dialplan fraud logic. Vendor-dependent. Often licensed as extra module.
Generic proprietary SBC only Good edge protection but no visibility into call logic fraud. SBC scales but backend not specified. Vendor lock-in. Custom rules require professional services contracts. Toll fraud happens in the call engine - SBC alone cannot block mid-call fraud. Depends on vendor edition.
Security by Design - Threat Control Matrix
Every known attack vector addressed at the correct layer.
Volumetric SIP DDoS
Millions of UDP/TCP packets overwhelming the signaling stack.
Optional anycast scrubbing upstream; UDP rate caps and connection limits at Kamailio edge before parsing.
SIP Flood / Scanning
High-rate INVITE or OPTIONS floods from botnets scanning for open SIP endpoints.
Kamailio pike module auto-bans IPs exceeding req/s threshold; entries written to htable; fail2ban drops at iptables.
Registration Attacks
Automated REGISTER floods targeting authentication endpoints to brute-force credentials.
Per-IP and per-AOR registration throttle; digest auth challenge; short ban windows on repeated failures.
Extension Enumeration
Attackers probe for valid extensions by analyzing 404 vs 401 response codes.
Uniform 401 Unauthorized returned for all extensions (valid and invalid); scanner User-Agent patterns blocked at edge.
Brute Force Auth
Credential stuffing against known extension ranges using leaked password lists.
htable failed-auth counter with escalating ban windows; fail2ban syncs block to OS firewall; per-account lockout.
Toll Fraud
Compromised extension dials premium international numbers generating large charges.
Per-account concurrent call caps; destination allow-lists; high-risk prefix blacklist; velocity limits; auto-suspend on anomaly.
RTP / Media Injection
Attacker injects forged RTP packets to hijack or disrupt active calls.
RTPengine anchors all media; strict source IP/port validation; SRTP-only on all legs; media ports firewalled except via anchored flows.
Eavesdropping
Passive interception of SIP signaling or RTP media on untrusted networks.
TLS for all SIP signaling; WSS for browser clients; SRTP for all media legs; DTLS-SRTP for WebRTC; zero cleartext on public side.
Topology Disclosure
SIP Via / Record-Route headers expose internal IP addresses to external parties.
Kamailio topology hiding module rewrites all headers; internal IPs never visible to endpoints or carriers.
Geographic Abuse
Traffic originating from regions where the platform has no legitimate users.
Geo-IP and ACL filtering at edge; only operating regions whitelisted; all other sources receive early termination.

Engagement Structure
Three phases from secure prototype to full production-grade handover. Each phase delivers a complete, independently valuable outcome.
Phase 1
Secure Architecture + Working Prototype
  • Kamailio SBC configured with anti-flood, auth challenge, topology hiding, and TLS/WSS termination
  • RTPengine running DTLS-SRTP to SRTP bridging with media anchoring
  • Asterisk PJSIP with per-tenant dialplan fraud guards, destination allow-lists, and velocity caps
  • coturn STUN/TURN for NAT traversal
  • Browser WebRTC softphone (JsSIP) registering over WSS and placing secure calls between clients
  • Containerized one-command stack deployable on any Ubuntu Linux server
Phase 2
Discovery, Validation & Extended MVP
  • Audit and integration review against existing infrastructure - identify gaps and integration points
  • Extended MVP proving secure SIP registration at scale under simulated load
  • Full WebRTC browser calling with validated DTLS-SRTP and ICE across NAT
  • Anti-fraud controls tuned to actual traffic patterns with real carrier trunk tests
  • Homer/HEP SIP capture, Prometheus metrics, and Grafana dashboards live
  • fail2ban integrated with Kamailio logs and synced to firewall rules
Phase 3
Documentation, Hardening & Handover
  • Full architecture diagrams in draw.io / Mermaid format, ready for internal documentation
  • Security hardening guide covering each layer with rationale and tuning parameters
  • Infrastructure-as-code repository with environment variable configuration and no hardcoded secrets
  • Runbook covering deployment, cert rotation, scaling procedures, and incident response
  • Test plan demonstrating every security control firing with expected log output
  • Knowledge transfer sessions covering operational procedures and architecture decisions
